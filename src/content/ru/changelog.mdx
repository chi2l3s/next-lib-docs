# Changelog

Этот changelog написан в инженерном формате: не только «что поменяли», но и «как это влияет на ваш код».

## Как читать изменения

Для каждого пункта полезно оценивать:

1. **Feature** — что добавлено.
2. **Behavior** — как изменилось поведение.
3. **Migration impact** — что нужно изменить в проекте.
4. **Risk** — что проверить до прод выката.

## 1.0.8

### Основные изменения

- Добавлен `NextLibContext` как единый runtime-контейнер.
- Усилен Dynamic Database слой: relations/include/upsert/auto migrations.
- Расширены платформенные возможности:
  - `Validation API`
  - `Reload API`
  - `Observability API`
  - `I18n API`
  - `Messaging API`
  - `Quests API`

### Изменения поведения

- В `DynamicTable` mutating операции с единичным контрактом (`update`, `delete`) более строго защищают от массовых accidental изменений.
- `findUnique(where)` валится, если найдено больше одной записи.
- `skip(offset)` в пагинации требует `take(limit)`.

### Влияние на интеграцию

Если вы переходите с более ранних билдов:

1. Проверьте, что GUI YAML использует актуальные snake_case ключи.
2. Проверьте, что DB сущности соответствуют текущим правилам маппинга.
3. Перенесите инфраструктурные bootstrap задачи в `NextLibContext`.
4. Добавьте централизованный reload пайплайн.

### Что протестировать после обновления

- startup sequence (`onEnable`)
- открытие всех GUI
- ключевые DB операции (create/read/update/delete/upsert)
- reload команды
- i18n fallback-сообщения

## 1.0.7

### Основные изменения

- Расширение query operators в Dynamic Database:
  - `>`, `<`, `>=`, `<=`
  - `LIKE`
  - `IN`
  - `BETWEEN`
  - starts/ends/contains helpers

### Практический эффект

- меньше ручного SQL;
- более выразительные репозитории;
- меньше шаблонного кода для фильтрации.

### Риски

- некорректное использование `null` с неподдерживаемыми операторами;
- `IN` без значений;
- `BETWEEN` с неверным количеством параметров.

## Рекомендованный протокол обновления

1. Сначала staging.
2. Бэкап данных.
3. Smoke test ключевых сценариев.
4. Прогон reload.
5. Проверка health/metrics после 10-15 минут нагрузки.
6. Только затем production rollout.

## Матрица проверки после апдейта

- **DB**: соединение, миграции, индексы, latency.
- **GUI**: загрузка yaml, клики, conditions/actions.
- **Commands**: регистрация, permissions, tab-complete.
- **I18n**: ключи и fallback.
- **Reload**: модульные отчёты.
- **Observability**: счетчики/таймеры/health.

## Change management рекомендации

- фиксируйте версию библиотеки явно;
- ведите внутренний changelog интеграции (что изменили у себя под апдейт);
- если меняете схемы БД, отдельно документируйте migration notes.

## Частые ошибки при обновлениях

- обновили артефакт, но не драйвер БД;
- обновили код, но оставили старые YAML ключи;
- пропустили reload тест и обнаружили проблемы уже на проде;
- не проверили контракт уникальности в `findUnique`.
